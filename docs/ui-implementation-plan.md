# UI Implementation Plan

最終更新: 2026-06-10

## 目的

新しいSvelte UIデザインを正とし、現在の「テキスト入力 -> LLM応答 -> irodori-TTS読み上げ -> 音声再生」の縦切りを壊さずに、未接続のUI要素を実処理へ段階的に接続する。

この計画書は、見た目の再設計ではなく、デザイン上すでに存在する操作・状態・表示をどの順番で実装するかを決めるためのもの。

## 前提

- 新UIの情報設計は採用する。
- Windows AMD / WSL構成とMacBookローカル構成の両方を維持する。
- 会話サーバーは引き続きLANまたはローカル限定で使う。
- MVPでは認証なし、会話履歴はメモリ保持、同時会話ターンは1つのみ。
- 既存の同期RESTテキスト会話は残し、音声入力フェーズでWebSocketを追加する。
- UIで「将来機能」と明示されている要素は、実装するまでは誤って効いているように見せない。

## 現状

### 実処理まで接続済み

- 会話サーバー接続先の編集とlocalStorage保存
- MacBookローカル用とdesktop PC接続用の接続先保存キー分離
- `/api/health` による会話サーバー / Ollama / irodori-TTS状態表示
- テキスト入力による `POST /api/turns/text`
- 送信直後の利用者発話表示
- 応答待ち表示
- 応答テキスト表示
- 生成音声URLの表示とカスタム再生ボタン
- 自動読み上げON/OFFのクライアント内保存と自動再生試行
- キャラクター名、キャラクター設定、読み上げ設定、話者の保存
- 設定保存時の会話履歴クリア
- 会話履歴クリア
- キャラクター画像アップロード
- native `<dialog>` による設定パネル
- MacBookローカル構成の `gemma4:e4b-mlx` 実接続

### UIはあるが処理未接続または限定的

- マイクボタン: 表示のみ。音声入力処理は未実装。
- 口調プリセット: localStorageに保存され、プレビュー表示のみ。会話サーバーには送っていない。
- 距離感スライダー: localStorageに保存され、ラベル表示のみ。会話サーバーには送っていない。
- 話す速さスライダー: localStorageに保存されるが、irodori-TTS speech requestへ渡していない。
- 読み上げ設定: サーバーに保存されるが、現行Irodori-TTS-Serverのspeech endpointへ直接渡していない。
- キャンセル: design notes上はMVP対象だが、現在の同期RESTテキスト会話にはキャンセルAPIがない。
- 応答生成の詳細状態: UIは「返答中」まで。Ollama生成中 / TTS生成中 / 音声準備中の分離はない。
- 長時間TTS生成: MacBookローカルでは初回生成が約127秒、以後も約46秒の実測があるが、UI上は単一の待機表示のみ。
- 参照音声登録: UIは話者選択に対応済みだが、実機 `/api/speakers` は `none` のみ。

## 実現可否

### そのまま実現可能

- 参照音声ファイルをIrodori-TTS-Serverへ登録し、`/api/speakers` へ反映する。
- MacBookローカルとdesktop PC接続のUI表示を、接続先に応じて出し分ける。
- 会話サーバーの依存状態エラーを、UI上で原因別に出す。
- 設定パネルのアクセシビリティ改善。
- E2Eのカバレッジ拡張。

### 小さなAPI拡張で実現可能

- 会話進行状態の細分化: `thinking` / `synthesizing` / `ready_to_play` などをサーバーまたはクライアントイベントとして表現する。
- TTS生成待ちのtimeout表示: サーバー側でtimeout種別を分け、UIに「読み上げ生成に時間がかかっています」を出す。
- 読み上げ速度: Irodori-TTS-Serverがspeech endpointで速度指定を受けられる場合のみ、会話サーバーのTTS requestへ橋渡しする。
- 口調プリセット / 距離感: サーバー保存設定へ組み込み、最終的な `character_prompt` へ合成する。

### 設計変更が必要

- 音声入力: ブラウザマイク、PCM変換、WebSocket、発話終端、STT、テキスト会話ターンへの接続が必要。
- キャンセル: 同期RESTだけでは十分ではない。最低でもクライアント側の結果破棄、理想的にはサーバー側ジョブID/キャンセルAPIまたはWebSocket制御が必要。
- 応答ストリーミング: Ollama token streaming、TTS分割、クライアント表示のイベント化が必要。MVP後に回す。
- 複数クライアント同時利用: 現状はbusy拒否。キューやセッション分離を入れるなら別設計。

## 実装フェーズ

### Phase 0: 計画と現在地固定

目的: 新デザインを実装の基準として固定し、未接続UIを明文化する。

作業:

- この計画書を追加する。
- `docs/handoff.md` に本計画への参照を追加する。
- 現行E2Eが新UIで通ることを維持する。

完了条件:

- `pnpm check`
- `pnpm build`
- `pnpm test:e2e`

### Phase 1: デザイン実装の仕上げ

目的: すでに動くテキスト会話UIを、新デザインとして破綻なく使える状態へ固める。

状態: 2026-06-10完了。同期RESTの制約上、返答生成中 / 読み上げ準備中の分離はクライアント側の待機時間による推定表示。

作業:

- [x] 設定パネルのフォーム説明を `aria-describedby` で入力に結びつける。
- [x] 設定パネルの保存中 / 接続中 / アップロード中状態を表示し、二重送信を防ぐ。
- [x] エラー表示を会話サーバー未接続、Ollama不可、irodori-TTS不可、TTS生成timeout、自動再生失敗に分ける。
- [x] MacBookローカルの長時間TTS生成に合わせ、待機表示を「返答生成中」「読み上げ準備中」へ分ける。
- [x] モバイル幅で設定ボタン、入力欄、スレッドスクロールが重ならないことを確認する。

実現可否:

- 既存APIの範囲で大半は実現可能。
- 生成状態の厳密な分離は、Phase 4のキャンセル/長時間処理設計と合わせると精度が上がる。

完了条件:

- E2Eに「設定パネルを開閉できる」「接続エラー時に原因が表示される」「自動読み上げOFF時はaudio autoplayしない」を追加する。
- Browserでdesktop幅とmobile幅のスクリーンショット確認を行う。

### Phase 2: 参照音声と読み上げ設定

目的: 「声 ・ 読み上げ」セクションを実体のある設定に近づける。

状態: 2026-06-10実装・手順整備済み。MacBookからdesktop PC / WSL構成へは会話サーバー経由で参照音声を登録できる。実機で `none` 以外の話者を使う確認は、参照音声ファイルの用意後に実施する。

作業:

- [x] Irodori-TTS-Serverの `voices/` に参照音声を登録する手順をdocsへ追加する。
- [x] MacBookから会話サーバー経由でIrodoriへ参照音声をアップロードするAPIとスクリプトを追加する。
- [ ] `/api/speakers` で `none` 以外が返ることを実機確認する。
- [ ] UIの話者選択保存後、次の会話ターンで選択話者が使われることを実機確認する。
- [x] `read_aloud_prompt` は現行speech endpointに直接渡せないため、「将来用メタデータ」として扱う表示を維持する。
- [x] Irodori側が速度指定を受けられるか確認し、可能なら `speed` をサーバー設定へ昇格する。不可ならプレビュー表示のまま残す。

実現可否:

- 話者選択は既存APIで実現可能。
- 読み上げ速度はIrodori-TTS-Serverの実API次第。未確認のままUIだけ効かせるのは避ける。

完了条件:

- 実機 `/api/speakers` に参照音声IDが出る。
- 設定保存後の読み上げで選択話者が使われる。
- docsにMacBookローカルとdesktop PC / WSLの参照音声配置先を明記する。

### Phase 3: 口調プリセットと距離感の実処理化

目的: 「性格 ・ 口調」セクションを、単なるローカルUIではなく会話応答に反映する。

作業:

- サーバー側 `AppSettings` に `tone_preset` と `distance` を追加するか、既存 `character_prompt` へ合成する設計を決める。
- 推奨は、保存データとして構造化値を持ち、LLM送信直前にsystem promptへ合成する方式。
- 既存の自由記述 `character_prompt` は残す。
- UIのプレビュー文は、選択値が会話に反映されることを示す内容へ更新する。
- 既存 `settings.json` の後方互換を保つ。

実現可否:

- Ollama側はsystem prompt変更で実現可能。
- ただし口調の安定性はモデル依存。UIで過度に厳密な制御を約束しない。

完了条件:

- サーバーテストで設定の保存/移行を確認する。
- E2Eでプリセット変更 -> 保存 -> 次ターンに設定が送られることをモックで確認する。

### Phase 4: 会話キャンセルと長時間処理の扱い

目的: 同期RESTのままでも、利用者が長い生成を待たされ続ける問題を緩和する。

状態: 2026-06-10クライアント側キャンセルを実装済み。同期RESTの制約上、キャンセルは画面上のpending turn破棄とfetch中断であり、サーバー側のLLM/TTS処理停止までは保証しない。

作業:

- [x] クライアント側キャンセルを先に実装する。`AbortController` でREST requestを中断し、結果を履歴に反映しない。
- [x] サーバー側で中断検知が難しい場合、キャンセルは「画面上の破棄」として明記する。
- [x] 500/timeout時の履歴非追加を維持する。
- [x] busy状態が残らないかサーバー側を確認する。`async with` lockにより正常終了・例外後は解放されることをテストで固定した。ただしクライアントキャンセル直後は、サーバー処理完了までbusyが続きうる。
- [ ] 将来のジョブID方式またはWebSocket方式へ移れるよう、会話進行状態の型を整理する。

実現可否:

- クライアント側キャンセルは実装可能。
- サーバー側の実処理停止は同期RESTだけでは限定的。

完了条件:

- 返答待ち中にキャンセルできる。
- キャンセルしたturnが履歴に残らない。
- サーバー側の処理完了または失敗後、busyが解除されて次の入力ができる。

### Phase 5: 音声入力

目的: デザイン上のマイクボタンを、音声から始まる会話ターンへ接続する。

作業:

- WebSocket APIを設計する。
- ブラウザでマイク権限を要求する。
- AudioWorkletまたはWeb AudioでPCM signed 16-bit little-endian / 16 kHz / monoへ変換する。
- 音声チャンクをWebSocket binary frameで送信する。
- サーバー側でVADを行い、発話終端を決める。
- faster-whisperでSTTを行う。
- 認識テキストを既存のテキスト会話ターンへ接続する。
- UI状態を `listening` / `thinking` / `synthesizing` / `speaking` / `error` へ拡張する。

実現可否:

- 技術的には実現可能。
- ただしWebSocket、音声処理、STT導入で影響範囲が大きいため、Phase 1-4の後に着手する。

完了条件:

- マイクボタンで音声セッションを開始できる。
- 発話終端後、認識テキストが会話履歴に表示される。
- AI応答と読み上げが既存テキストターンと同じ形で返る。
- MacBookローカルとdesktop PC / WSLのどちらか一方で実機成功し、その後もう一方で確認する。

### Phase 6: Tauriとスマホ

目的: Webクライアントの安定後に配布形態と実端末対応へ進む。

作業:

- PC WebでPhase 1-5が安定してからTauriを追加する。
- Tauriでは会話サーバーを同梱起動するか、既存の接続先指定方式を維持するかを再検討する。
- スマホ実機ではLAN接続、マイク権限、AudioWorklet互換性、自動再生制限を確認する。

実現可否:

- Tauri化は可能だが、会話サーバー管理方針を決めないと中途半端になる。
- スマホは音声入力実装後に検証すべき。

## 優先順位

直近の推奨順:

1. desktop PC側WSLリポジトリへPhase 2/4の修正を反映し、会話サーバー再起動後に `/api/settings` の `speech_speed` を実機確認する
2. 参照音声ファイルを用意し、Phase 2の話者実機確認を完了する
3. Phase 4残作業: 将来のジョブID/WebSocket方式に向けた状態型整理
4. Phase 3: 口調プリセットと距離感の実処理化
5. Phase 5: 音声入力
6. Phase 6: Tauriとスマホ

理由:

- まず既に見えているUIを「操作しても破綻しない」状態へ固める必要がある。
- 声質はこのアプリの体験価値に直結するため、音声入力より先に参照音声を通す。
- MacBookローカルIrodoriの生成時間が長いため、キャンセルと待機表示は早めに必要。
- 口調プリセットは魅力的だが、LLM制御の品質検証が必要なので、声と待機体験の後に回す。
- 音声入力は最も大きな機能追加なので、現在のテキスト会話体験を安定させてから入る。

```bash
cd server
UV_CACHE_DIR=/private/tmp/uv-cache-gemma4-irodori uv run pytest
```

```bash
cd client

```

実サービス確認:

```bash
./scripts/mac/check-mac-stack.sh

```

実サービス確認は、変更内容に応じてMacBookローカルまたはdesktop PC / WSLのどちらかを先に行い、リリース候補では両方を確認する。

## 既知のリスク

- MacBookローカルIrodoriはCPU/MPS混在で生成が遅い。UI上の待機説明がないと失敗に見える。
- desktop PC / WSL構成はLAN IP、Windows Firewall、portproxyに依存する。
- irodori-TTSのOpenAI互換speech endpointがどの詳細パラメータを受けるか未確定。
- 自動再生はブラウザ制限により常に成功するとは限らない。
- native `<dialog>` は基本的に妥当だが、Safariやモバイルでライトディスミス、スクロール、フォーカス復帰を実機確認する必要がある。
- 音声入力はマイク権限、ブラウザ互換性、VAD/STTの遅延が主要リスク。
